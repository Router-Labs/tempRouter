Why

Payment was the easy half.

MPP already does per-chunk stablecoin billing. tempRouter ships the half MPP left open — provable confidentiality — as a payable endpoint any agent can discover and pay.

Verify before you pay

The agent runs Intel DCAP on the live enclave quote before signing the first voucher. Fail the check, sign zero vouchers — and a bad enclave can't decrypt the prompt anyway.

Two-phase attestation

Pre-pay: genuine TDX + UpToDate TCB + key binding + code measurement. Post-pay receipt: enclave ed25519 over the exact ciphertext. Every check is auditable on the wire.

Blind relay

The prompt is E2E-encrypted (Arcium RescueCipher + X25519) to a key that only exists inside the enclave. tempRouter holds no key and never sees plaintext — it forwards ciphertext.

Client-side secret policy

Your agent detects API keys, private keys, JWTs and PII locally and forces the private lane — the only correct place, since a blind relay can't classify what it can't read.

Per-chunk MPP metering

An MPP session over SSE: pay per response-chunk in pathUSD, settled in ~2 on-chain txs. One charge per inference by default; multi-chunk metering supported.

Discoverable + payable

OpenAI-shaped request, MPP 402 flow, and a machine-readable discovery doc so any agent or registry (mpp.dev/services, MPPScan) can find and pay the endpoint.

Architecture

Three parties, one blind hop.

Plaintext is only ever seen by your agent and the attested enclave. tempRouter sits in the middle and forwards ciphertext — it cannot read the prompt or the answer.

agent (your process) │ detectSensitive → verify (DCAP) → encrypt → pay → decrypt │ holds the wallet + the only decryption key ▼ ciphertext + MPP vouchers tempRouter (this service — BLIND RELAY, holds no key) │ mppx session: 402 → pay-per-response-chunk → SSE │ forwards ciphertext, meters the reply, never sees plaintext ▼ ciphertext Phala Intel TDX enclave (SolRouter prod · chain-agnostic) decrypts INSIDE hardware · runs gpt-oss:20b · re-encrypts returns a TDX-attested receipt (ed25519 over the ciphertext)
Use Cases

When to route private.

tempRouter is the confidential lane — use it when the prompt must not leak. For everything else, any public model is cheaper and smarter.

Use tempRouter when:

  • ✓Prompt contains API keys (sk-…, sk-ant-…)
  • ✓Includes private keys (0x… 64-hex), seed phrases, wallets
  • ✓Holds JWTs, session tokens, passwords
  • ✓Contains PII / customer data — names, emails, SSNs
  • ✓Proprietary code or business logic
  • ✓Legal, medical, or financial records (GDPR / HIPAA / SOC2)

Don't use tempRouter when:

  • ✗General questions — "how does X work?"
  • ✗Creative writing — stories, poems
  • ✗Casual chat or open-ended conversation
  • ✗Non-sensitive coding help — public APIs, tutorials
  • ✗Public research — lookups, summaries
The private lane runs an OSS model (gpt-oss:20b) — it's the confidential lane, not a frontier-model replacement.

Flow

How an agent uses it.

  1. Detect — your agent's policy flags a secret/PII payload and forces the private lane. Non-sensitive prompts go to any public model (out of scope).
  2. Verify — GET /tee/attestation, run Intel DCAP on the quote. Fail → sign zero vouchers, refuse to pay.
  3. Encrypt — encrypt the prompt to the enclave's X25519 key. Plaintext never leaves the agent.
  4. Pay — POST /v1/chat/completions/stream → 402 → open an MPP session on Tempo, pay per response-chunk in pathUSD.
  5. Decrypt — reassemble the ciphertext chunks and decrypt locally. Only the agent and the attested enclave ever see plaintext.
Integrate

Drop it into your agent.

Infrastructure, not a destination. The same verify-before-pay lane ships at four altitudes — each does the whole dance: verify → encrypt → pay per chunk → decrypt. Pick the one that fits.

// @temprouter/sdk — the whole lane in one call
import { TempRouter, detectSensitive } from '@temprouter/sdk'

const client = new TempRouter({ serverUrl: 'https://mpp.solrouter.com', account }) // funded Tempo wallet

if (detectSensitive(prompt).sensitive) {
  const { answer, paid } = await client.infer(prompt) // verify → encrypt → pay → decrypt
}

Trust, but verify

What your agent actually checks.

Nothing is taken on trust. Before any voucher is signed, the agent runs every check below and prints a per-check PASS/FAIL report.

✓Intel DCAP cert chain. The quote is genuine Intel TDX hardware (PCK chain → Intel root + TCB UpToDate).
✓Key binding. The quote's report_data commits to the enclave's X25519 + ed25519 keys.
✓Enclave signature. The in-enclave ed25519 key signed this exact ciphertext (post-pay receipt).
✓Quote self-consistency. The enclave committed to its own quote digest.
✓Measurement. mrtd/rtmr compared to the boot-advertised value (soft-pin); strict-pin opt-in.
✓Fail-closed. Any failure → zero vouchers signed, no payment, by construction.

Honest scope: the private lane runs an OSS model (gpt-oss:20b) inside the enclave — use it for confidential payloads, not as a frontier-model replacement. Verification covers DCAP cert-chain + key binding + enclave signature; code-measurement pinning is opt-in. The enclave key is attached to the MPP session as a settlement label, not an enforced gate.
